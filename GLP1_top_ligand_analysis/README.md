# GLP-1R Activation Scoring Workflow

## Overview

This repository implements a structure-based workflow to estimate whether a candidate small molecule preferentially stabilizes the active state of GLP-1R relative to the inactive state.

The current study configuration of record is the 715-compound topological-screen rerun defined in [configs/study_715_topological.yaml](configs/study_715_topological.yaml). The workflow uses an experimental receptor ensemble, Meeko-based receptor and ligand preparation, AutoDock Vina docking, and bootstrap aggregation of active-versus-inactive docking scores.

For a lighter-weight run that keeps the same receptor ensemble but trims ligand-state expansion and docking replicates, use [configs/study_715_topological_lite.yaml](configs/study_715_topological_lite.yaml). It reduces the tautomer cap from 8 to 4, uses 2 Vina seeds instead of 3, lowers bootstrap iterations from 3000 to 1000, and is intended to cut the total docking workload by roughly half.

This is a docking-tier activation-preference analysis. It is not a rigorous free-energy calculation. The score should be interpreted as a state-preference docking metric that prioritizes compounds for higher-fidelity MD, MM/GBSA, and FE work.

## Scientific Objective

The analysis is designed to answer a specific question:

Which ligands dock more favorably to active-state GLP-1R models than to inactive-state GLP-1R models?

The workflow does not compare raw total system energies of the receptor structures. Instead, it compares docking-derived binding surrogates across receptor states.

### Primary Score

For each ligand, the workflow computes:

`state_preference_score = median(score_inactive) - median(score_active)`

where more negative Vina scores represent more favorable docking.

Interpretation:

- Positive `state_preference_score`: the ligand docks more favorably to active-state structures than to inactive-state structures.
- Near-zero `state_preference_score`: no clear docking preference.
- Negative `state_preference_score`: the ligand docks more favorably to inactive-state structures.

### Exported Per-Ligand Score

The main exported score column is:

- `activation_preference_score`

with uncertainty columns:

- `activation_preference_ci_low`
- `activation_preference_ci_high`

For backward compatibility, the same values are also exported under:

- `final_publication_score`
- `final_score_ci_low`
- `final_score_ci_high`

The confidence interval is generated by bootstrap resampling across the collected active and inactive docking scores for that ligand.

## Orthosteric Site Definition

The orthosteric site is defined from the peptide-bound reference structure `6X18_GLP1_GLP1R.pdb`.

The workflow:

1. Identifies GLP-1R residues within 4.5 A of the bound peptide.
2. Stores those residues in `derived.../pocket/orthosteric_site.json`.
3. Computes a ligand bounding box from the peptide coordinates.
4. Uses that reference site to map receptor-specific docking box centers across the ensemble.

This is important scientifically because the active and inactive receptor structures are not assumed to share the same raw Cartesian frame. Instead of blindly reusing the `6X18` ligand coordinates, the workflow computes a receptor-specific docking center by:

1. defining the orthosteric residue set in the reference structure,
2. computing the reference residue-box center,
3. measuring the offset between the peptide box center and the residue-box center in `6X18`,
4. applying that offset to the homologous residue-box center in each receptor structure.

This prevents the inactive-state docking box from drifting away from the orthosteric pocket.

## Receptor Preparation

Receptor preparation is performed by `prepare-receptors` and `prepare-pdbqt`.

### Receptor-Only Extraction

For each configured structure, the workflow:

- identifies the GLP-1R chain,
- writes a receptor-only PDB,
- preserves peptide-proximal waters only for the `6X18` reference when those waters are part of the orthosteric-site definition,
- records missing residues and alignment statistics in the receptor manifest.

### Alignment

Non-reference receptors are aligned to the `6X18` frame using the orthosteric residue set. The alignment is used to normalize the receptor-only PDB coordinates and to document how closely each structure maps onto the reference pocket.

### PDBQT Generation

PDBQT generation uses Meeko when available. This is the preferred path because:

- receptors are written as rigid receptor PDBQT files suitable for Vina,
- ligands receive AutoDock-style atom typing and charge assignment,
- the invalid receptor torsion-tree issue seen with naive Open Babel receptor conversion is avoided.

If Meeko is unavailable, ligand conversion can fall back to Open Babel, but the intended workflow is Meeko-based.
The fallback now streams multi-molecule SDF files through the Open Babel Python bindings and writes one PDBQT per ligand microstate title.

## Ligand Preparation

Ligand preparation is performed from the configured screening CSV.

For the 715-compound study, the input file is:

- `screening_full_1M_topological_hashed.csv`

The workflow performs:

- exact use of `zinc_id` as the ligand identifier,
- exact use of `smiles` as the input structure,
- standardization and canonicalization,
- protonation-aware and tautomer-aware expansion near assay pH 7.4,
- up to 2 microstates per ligand,
- up to 8 tautomers per microstate,
- 3D conformer generation,
- geometry optimization,
- PAINS and reactive-group flagging,
- Murcko scaffold assignment for downstream shortlist diversity control.

The prepared library is stored as:

- a CSV manifest,
- an SDF file,
- a PDBQT directory after `prepare-pdbqt`.

For the current 715-compound study, the prepared set contains 715 unique ligands and multiple dockable microstates per ligand.

## Docking Protocol

### Engine

The implemented docking engine is AutoDock Vina.

Current parameters from the active study config:

- `exhaustiveness: 24`
- `num_modes: 20`
- `seeds: [11, 29, 47]`

### Ensemble Expansion

Each ligand microstate is docked against:

- 3 active structures
- 2 inactive structures
- 3 random seeds

This yields 15 docking attempts per ligand microstate.

### Job Table

The complete docking workload is written to a job table under:

- `.../docking/<study>_jobs.csv`

Each row stores:

- `job_id`
- receptor state and structure identifiers
- ligand and microstate identifiers
- receptor and ligand PDBQT paths
- docking box center and size
- exhaustiveness, number of modes, and seed

### Resume Behavior

The docking runner supports `--resume`.

Important implementation detail:

- completed `job_id`s are read from the existing result CSV,
- pending jobs are filtered after that,
- `--max-jobs` is applied to the pending list, not the original list,
- output rows are de-duplicated by `job_id` before writing.

This makes the multi-day batching strategy safe for local and HPC use.

## Score Aggregation and Comparison

### Per-Ligand Aggregation

After docking, `score-docking` groups rows by `ligand_id` and computes:

- `n_active`
- `n_inactive`
- `active_median_score`
- `inactive_median_score`
- `state_preference_score`
- `state_preference_ci_low`
- `state_preference_ci_high`

The bootstrap CI is based on repeated resampling of the active and inactive score sets for that ligand.

### Activation Preference Table

`synthesize-publication-score` merges the docking summary with the ligand metadata and writes the per-ligand activation table.

For the current local docking-tier workflow:

- `score_tier = docking` for ligands with a valid state-preference score,
- `score_tier = unscored` for ligands that do not yet have both active and inactive coverage,
- `activation_preference_score` is the operative final score.

### Comparison Against Screening

`compare-screening` compares `weighted_coverage_pct` from the original pharmacophore/topological screen against `activation_preference_score`.

The report includes:

- Spearman correlation with bootstrap confidence interval
- Pearson correlation with bootstrap confidence interval
- Kendall tau with bootstrap confidence interval
- top-k overlap
- Jaccard overlap
- NDCG

This comparison is only scientifically interpretable once the docking run is complete for the full ligand set. Partial docking coverage will produce partial score tables and can yield undefined correlation metrics.

## How To Run the Analysis

### One-Time Setup

Check the available external tools:

```powershell
python scripts/check_external_tools.py
```

The intended runtime environment should provide:

- Python
- Meeko
- Vina
- Java
- Open Babel

### Core Pipeline Commands

For the 715-compound study:

```powershell
python scripts/pipeline.py --config configs/study_715_topological.yaml extract-pocket
python scripts/pipeline.py --config configs/study_715_topological.yaml prepare-receptors
python scripts/pipeline.py --config configs/study_715_topological.yaml prepare-ligands --mode screening
python scripts/pipeline.py --config configs/study_715_topological.yaml prepare-pdbqt --libraries screening
python scripts/pipeline.py --config configs/study_715_topological.yaml build-jobs --library screening
```

After docking has completed:

```powershell
python scripts/pipeline.py --config configs/study_715_topological.yaml score-docking --library screening --input derived_715_topological\docking\glp1r_activation_scoring_715_topological_vina_results.csv
python scripts/pipeline.py --config configs/study_715_topological.yaml synthesize-publication-score --library screening --docking derived_715_topological\scoring\glp1r_activation_scoring_715_topological_ranked.csv
python scripts/pipeline.py --config configs/study_715_topological.yaml compare-screening --library screening --publication derived_715_topological\scoring\glp1r_activation_scoring_715_topological_publication_scores.csv
python scripts/pipeline.py --config configs/study_715_topological.yaml export-md-manifest --library screening
```

### Local Multi-Day Batch Runner

For Windows local execution, use:

```powershell
powershell -ExecutionPolicy Bypass -File scripts\run_screening715_local.ps1
```

This wrapper:

- reads the full 715-study job table,
- runs Vina in 1000-job batches,
- prints batch progress in the terminal,
- resumes from prior completed jobs,
- refreshes the ranked score and comparison tables when docking finishes.

### HPC Usage

Do not use the PowerShell wrapper on Linux HPC.

Use the Python pipeline directly from your scheduler script, for example:

```bash
python scripts/pipeline.py --config configs/study_715_topological.yaml run-vina-jobs \
  --library screening \
  --jobs-csv derived_715_topological/docking/glp1r_activation_scoring_715_topological_jobs.csv \
  --max-jobs 1000 \
  --cpu 1 \
  --workers 8 \
  --resume
```

Submit repeated or array-based jobs until the result CSV reaches the full job count. Then run the post-docking aggregation commands.

## Current Scientific Boundaries

### What This Workflow Does Well

- Uses an explicit active versus inactive receptor ensemble
- Prevents coordinate-frame errors in the docking box
- Tracks uncertainty via bootstrap state-preference intervals
- Preserves ligand identity for direct comparison to `weighted_coverage_pct`
- Produces ranked handoff manifests for later MD and FE refinement

### What This Workflow Does Not Yet Claim

- It does not produce rigorous binding free energies
- It does not model membrane relaxation during local docking-tier scoring
- It does not establish efficacy, potency, or signaling bias experimentally
- It does not justify mechanistic claims from docking alone

Accordingly, `activation_preference_score` should be used as a prioritization metric, not as a definitive physical free energy of activation.
